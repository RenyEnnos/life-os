# Plan: 00-02 Integrated Error Handling

**Phase:** 00 - Fixes
**Plan:** 00-02
**Status:** Ready to execute

## Goal
Integrate the `AppError` class with the centralized `errorHandler.ts` to provide a consistent error reporting and tracking system across the app.

## Context
- `src/shared/errors/index.tsx` defines `AppError`.
- `src/shared/lib/errorHandler.ts` uses `ApiError` but not `AppError`.
- `http.ts` and `api.ts` patterns vary across features.

## Task Breakdown
1. **Unify Error Classes**: Update `errorHandler.ts` to accept `AppError` and handle its properties (e.g., category, severity).
   - Files: `src/shared/lib/errorHandler.ts`, `src/shared/errors/index.tsx`
2. **Standardize API Client**: Ensure `fetchJSON` and its wrappers in `http.ts` properly catch, wrap, and re-throw errors using the new unified handlers.
   - Files: `src/shared/api/http.ts`
3. **Audit Feature API Logic**: Quickly check `habits.api.ts`, `tasks.api.ts`, and `auth.api.ts` to ensure they aren't suppressing errors but letting them bubble up for global handling.
   - Files: `src/features/*/api/*.api.ts`

## Verification
- [ ] Forcing a 500 error in a fetch call results in a high-severity `AppError` log.
- [ ] UI components receive standardized error messages for toasts.
- [ ] App doesn't crash on unhandled API errors but shows a generic fallback message.
