# Plan: 04-01 Status Migration & Backend Update

**Phase:** 04 - Task Management (Kanban)
**Plan:** 04-01
**Status:** Ready to execute

## Goal
Migrate the task data model from a boolean `completed` flag to a multi-state `status` field to support the Kanban board.

## Context
- `Task` currently uses `completed: boolean`.
- Backend APIs need to be updated to accept and store the `status` field.
- Database migration is required to add `status` to the `tasks` table.

## Task Breakdown
1. **DB Migration**: Create a SQL migration to add `status` (ENUM: 'todo', 'in-progress', 'done') and map existing data.
   - Files: `supabase/migrations/20260302000001_add_task_status.sql`
2. **Frontend Types**: Update `shared/types.ts` and `src/features/tasks/types.ts` to include `status` and keep `completed` as a derived/deprecated field for backward compatibility.
   - Files: `shared/types.ts`, `src/features/tasks/types.ts`
3. **API Logic**: Update `tasks.api.ts` to include `status` in the list, create, and update methods.
   - Files: `src/features/tasks/api/tasks.api.ts`
4. **Backend Route**: Ensure the Express backend (if applicable) or Supabase RLS allows the new `status` field.
   - Files: `api/routes/tasks.ts` (if exists)

## Verification
- [ ] Task query returns the `status` field correctly for existing records.
- [ ] Creating a new task sets the default status to 'todo'.
- [ ] Updating a task status persists correctly in the database.
