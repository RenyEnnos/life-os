# Plano 01-01: Padronização de Schemas e Refatoração de APIs

## 🎯 Objetivo
Unificar a verdade dos dados através de schemas Zod compartilhados, eliminando validações manuais e garantindo consistência entre Frontend e Backend.

## 🧱 Must-Haves
- Schemas centralizados em `src/shared/schemas`.
- Mapeamento de erros Zod para mensagens amigáveis (i18n-ready).
- Zero validações manuais redundantes em `habits.api.ts`, `tasks.api.ts` e `finances.api.ts`.
- Backend (`api/`) consumindo schemas via aliases corretos.

## 📋 Tarefas

<tasks>
- [ ] **Configuração de Paths**: Atualizar `tsconfig.json` para garantir que o backend possa importar de `src/shared/schemas` sem erros de resolução.
- [ ] **Consolidação de Schemas**: Mapear campos divergentes (ex: `goal` vs `target_value` em Habits) e unificar em schemas canônicos.
- [ ] **Implementação do Error Formatter**: Criar utilitário `formatZodError.ts` para converter o array de problemas do Zod em strings legíveis por humanos.
- [ ] **Refatoração de APIs (Frontend)**: Substituir utilitários `validate...` em `src/features/*/api` pelo middleware de validação Zod.
- [ ] **Refatoração de Rotas (Backend)**: Aplicar o middleware de validação nas rotas críticas do Express.
</tasks>

## 🧪 Verificação (DoD)
- [ ] `npm test` passa para todos os schemas unificados.
- [ ] Tentativa de criar entidade inválida via UI exibe mensagem clara (ex: "O nome é obrigatório" em vez de "Required").
- [ ] Requisição via Postman/Insomnia para o backend com dados inválidos retorna 400 Bad Request com detalhes do Zod.
