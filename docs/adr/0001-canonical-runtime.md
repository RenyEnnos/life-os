# ADR-0001: Canonical runtime and release evidence

Status: provisional
Date: 2026-07-16
Decision owner: repository maintainer
Operational author: Codex under the autonomous recovery mandate
Related issues: #82, #83, #85, #86, #87, #88, #89
Supersedes: none
Superseded by: none

## Context

LifeOS contains multiple runtime narratives:

- `README.md` and `docs/mvp/canonical-mvp.md` describe an invite-only weekly loop in React and Express;
- `AGENTS.md` previously froze runtime selection;
- `.github/workflows/ci.yml` builds web and Electron and uses Electron smoke as a release gate;
- `src/features/mvp/api/mvp.api.ts` selects HTTP or IPC at runtime;
- Electron, PWA and incomplete Android scripts remain in the repository;
- JSON, Prisma/Postgres, SQLite and Supabase participate in different paths.

The product decision in `docs/product/2026-07-16-product-positioning-brief.md` narrows the current product to the weekly loop for invited design partners. No validated user requirement establishes offline desktop installation, mobile packaging or intentional multi-runtime parity as necessary today.

This ADR records an operationally active but reversible decision. It is not represented as maintainer ratification. The maintainer may accept, reject or supersede it later.

While provisional, it takes operational precedence for runtime classification over conflicting desktop-first release documents and broad-suite historical plans. Those documents remain present until #89 classifies or reconciles them.

Cross-runtime facts:

- `src/features/mvp/api/mvp.api.ts` implements the same renderer-facing operations over HTTP or IPC, but no parity test proves identical identity, error or persistence behavior.
- Web user IDs are generated by the file-backed auth repository or mapped into Prisma; desktop user IDs come from Supabase/local sessions. ID compatibility across stores is not established.
- `src/config/routes/access.ts` permits invite access through profile/invite state, the explicit bypass flag, development mode or localhost. Development mode, localhost and the relevant flag also participate in administrative gating under their respective controls. These are demo/internal controls, not a published authorization model.
- The file-backed auth repository seeds a known local fallback invite when no invite config is supplied. #87 must prevent that behavior from becoming a production boundary.

## Decision drivers

- match the runtime to the product that is actually routed and documented;
- define one authoritative release target;
- minimize operational load for an AI-native solo-maintained project;
- avoid deleting desktop data paths before migration evidence exists;
- unblock security and CI decisions without a big-bang rewrite;
- preserve a clean rollback path while evidence remains incomplete.

## Options considered

### A. Web-first

React and Express are canonical. HTTP is the official transport. Browser/API evidence becomes authoritative. Electron is experimental until either promoted by evidence or retired through scoped migration work.

Benefits: aligns with the canonical MVP, default development path and product-positioning decision; one primary deployment boundary.
Costs: browser E2E and web release evidence must be restored; Electron-specific paths remain temporarily.
Risks: file-backed defaults and web authentication are not yet publication-ready.

### B. Electron-first local

Electron, IPC and local persistence become canonical; web becomes preview or legacy.

Benefits: existing packaged smoke and local/offline infrastructure.
Costs: installer, updates, signing, local migration and desktop support become mandatory.
Risks: no validated user requirement justifies that cost; current product docs and default flow would need reversal.

### C. Intentional multi-runtime product

HTTP and IPC are supported adapters with parity and separate release matrices.

Benefits: maximum channel flexibility.
Costs: duplicated auth, data, packaging, security and test obligations.
Risks: highest maintenance burden and greatest chance that agents validate the wrong path.

### D. Web canonical with an indefinitely retained desktop path

Web is named canonical, but Electron remains in CI and release language without an explicit experimental contract.

Benefits: smallest immediate diff.
Costs: preserves the contradiction that caused #85.
Risks: “temporary” dual authority becomes permanent.

## Ranking

1. **A. Web-first** — recommended because it matches the validated product boundary with the lowest long-term operational surface.
2. **B. Electron-first** — second best coherent end state if design-partner research proves installed offline use essential; it is preferable to ambiguous dual authority but currently lacks product evidence.
3. **C. Intentional multi-runtime** — coherent but disproportionate without validated need for both channels.
4. **D. Indefinite desktop retention** — acceptable only as a short transition tactic, not an end state, because it preserves the contradiction.

## Decision

Adopt **Option A: Web-first** provisionally.

### Canonical product runtime

- client: React/Vite in a browser;
- server: Express;
- transport: HTTP for auth and `/api/mvp/*`;
- distribution: web deployment; exact hosting/container contract is decided by #86;
- authoritative end-to-end evidence: browser execution of the invite/auth boundary and complete weekly loop against the supported server/persistence fixture.

### Runtime classification

| Surface | State | Contract during transition |
|---|---|---|
| Browser + Express + HTTP | **canonical** | Required release target |
| PWA/service worker | experimental | Build may remain; no offline/readiness claim until separately validated |
| Electron + IPC | experimental | Retain without release authority; no deletion before data/consumer audit |
| Android/Capacitor | unsupported | Scripts do not establish support; do not run or advertise |
| Broad legacy feature routes | legacy/hidden | Not part of the canonical product |

### Persistence classification

Runtime selection does not silently choose a permanent database.

| Persistence | Current role | State under this ADR |
|---|---|---|
| File-backed JSON MVP repository | Default web/demo repository | Transitional; local/demo only until #87 defines publication boundary |
| Prisma/Postgres MVP repository | Optional server repository | Candidate for supported web persistence; requires live migration/rollback evidence |
| SQLite | Electron/local data and legacy IPC | Experimental-runtime data; preserve until migration/retention decision |
| Supabase | Desktop session/sync paths and RLS test environment | Not canonical persistence by implication; #87 defines identity/data boundary |

## Data, identity and migration risks

### Web file-backed to Prisma/Postgres

- User records live in `.data/auth-state.json`; MVP workspaces are keyed by those user IDs in `.data/mvp-workspace.json`.
- Prisma has its own user and relational MVP schema. The repository contains no proved import that preserves every file-backed ID, invite claim, plan/action ID, timestamp and event ordering.
- Before any cutover: freeze writes or take an atomic snapshot, back up both JSON files, validate schema/version, map users by stable ID rather than email alone, import into a disposable database, compare counts and representative snapshots, then run the complete loop.
- Rollback: retain untouched JSON backups and the prior application version; do not permit dual writes unless a conflict policy and reconciliation test exist.

### Electron JSON/SQLite/Supabase to web

- Desktop MVP JSON is separate from default web JSON and is keyed by the desktop session user ID.
- Desktop auth tokens and user identity live in SQLite `auth_session`; the SQLite database also owns legacy feature records. Supabase identity may share a user ID, while local fallback sessions may not.
- A migration must inventory each Electron `userData` location, distinguish local and Supabase identities, export MVP JSON independently from legacy SQLite, map IDs explicitly, and never copy access/refresh tokens into web auth state.
- Token migration is prohibited: users must reauthenticate or claim a controlled account link. Secrets and encrypted tokens are not portable credentials.
- Rollback: keep the original desktop files read-only, retain a compatible export reader, and verify that failed imports do not partially claim invites or overwrite a web workspace.

### Compatibility and loss controls

- Define one owner for each record during migration; never let web and Electron both be authoritative.
- Record source runtime, source user ID, target user ID, schema version and import checksum in a migration ledger.
- Detect collisions for user, plan, action, reflection and event IDs before writes.
- Validate counts plus semantic invariants: one active user mapping, confirmed plans retain actions/status, timestamps remain ordered, feedback ratings remain bounded, and deleted/reset state is not resurrected.
- Provide export, dry-run, backup, atomic commit where possible, and tested rollback before changing a runtime from experimental to archived/removed.

## Release contract

A release claim for the canonical runtime requires all of the following:

1. frozen dependency installation with the repository lockfile;
2. typecheck and lint with no errors;
3. configured unit and integration tests with documented secrets/fixtures;
4. web production build;
5. server build and startup against the supported non-production persistence fixture;
6. authoritative browser E2E covering invite registration/login, onboarding, plan generation and confirmation, daily action/check-in, reflection and feedback;
7. rollback instructions for the deployed server and data migration involved;
8. no unresolved critical security or data-integrity finding for the chosen operating mode.

Until #86 implements and proves this contract, repository checks are development evidence, not proof of a releasable web product.

Electron build or smoke may remain advisory during transition, but cannot block or prove the canonical web release. Conversely, browser evidence does not prove Electron works.

## Consequences

### Positive

- product, runtime and transport point to one primary path;
- #87 can define demo versus published web boundaries;
- #86 can align CI and Docker with one release target;
- agents no longer infer desktop authority from historical smoke tests;
- Electron data is preserved while its future is decided safely.

### Negative

- current CI and release documentation remain temporarily inconsistent;
- browser E2E is not presently authoritative or fully active;
- two transport adapters remain until explicit cleanup work is ready;
- the web path still needs an approved persistence and security mode.

### Risks

- maintainers may interpret “web-first” as authorization to delete Electron;
- file-backed persistence may be deployed beyond its safe operating mode;
- PWA generation may be mistaken for offline support;
- CI may appear green while the authoritative browser contract is absent.

Controls: explicit classifications above, #86/#87 gates, and prohibition on runtime removal without a ready issue.

## Maintainer decisions and ratification

The autonomous mandate authorizes the provisional Option A decision so recovery can continue. It does not claim that the maintainer personally selected A.

The maintainer may later choose:

1. ratify A and change this ADR to `accepted`;
2. select B if installed offline use is validated, requiring a superseding ADR before web removal;
3. select C and explicitly fund the parity/security/release cost;
4. reject all options and provide a new product/runtime constraint;
5. keep A provisional until #86/#87 evidence exists.

Additional decisions that remain outside this ADR:

- publication persistence: file-backed demo versus Prisma/Postgres or another approved store (#87);
- Supabase role in web identity/data (#87);
- hosting/container/update and required checks (#86);
- whether PWA is supported or removed (#86/follow-up);
- retention, export or removal of Electron and Android artifacts (ready implementation issues after inventory);
- acceptance or supersession of this ADR (maintainer).

## Migration

1. Record this ADR and remove only the runtime-selection freeze.
2. #87 defines supported web operating modes, identity, secrets and persistence boundaries.
3. #86 maps checks to commands, restores authoritative browser E2E, and reconciles Docker ports/startup.
4. #89 reconciles README, release ladder, desktop documents and historical PRDs with this ADR.
5. Create scoped issues for Electron archival/removal only after consumer, stored-data and rollback evidence exists.
6. Reassess PWA separately; remove Android scripts only through a ready cleanup issue.

No runtime code, adapter, schema or stored data changes in this decision.

## Rollback and supersession

Rollback trigger:

- validated design-partner research demonstrates that installed offline desktop use is essential; or
- the supported web boundary cannot meet security, privacy, reliability or operating-cost requirements; or
- the maintainer rejects the provisional decision.

Rollback action:

1. leave HTTP and IPC adapters intact during evidence gathering;
2. supersede this ADR with Electron-first or intentional multi-runtime status;
3. rewrite product/release authority before changing code;
4. require packaged Electron, update and local-data migration evidence before calling desktop supported.

## Validation

The decision is operationally validated when:

- #87 records a coherent publishable or explicitly local/demo web mode;
- #86 produces a green canonical web release lane with browser E2E;
- #89 removes contradictory runtime authority from active documentation;
- no required Electron behavior or stored data was removed without migration evidence;
- the maintainer accepts or supersedes the provisional ADR after reviewing those results.

Until then, status remains `provisional`.
